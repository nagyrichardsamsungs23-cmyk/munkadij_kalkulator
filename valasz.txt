A legjobb felépítés erre: **webes alkalmazás + Android app ugyanabból a kódbázisból**. Így nem kell külön weboldalt és külön Android appot fejleszteni.

## 1. Javasolt tech stack

**Ajánlott MVP stack:**

1. **Frontend:** Next.js + React + TypeScript
   Ez adja a webes felületet. A Next.js támogatja a PWA jellegű működést, amely app-szerű élményt ad böngészőben és telefonon is. ([nextjs.org][1])

2. **Android app:** Capacitor
   A Capacitor webes alkalmazást tud natív Android alkalmazásként csomagolni. Így ugyanaz a felület fut a weboldalon és Androidon is. ([Capacitor][2])

3. **Backend és adatbázis:** Supabase
   A Supabase PostgreSQL adatbázist, hitelesítést, fájltárolást, jogosultságkezelést és szerveroldali függvényeket ad egy rendszerben. ([Supabase][3])

4. **Adatbázis:** PostgreSQL
   Ez jó választás, mert a munkafolyamatok, árak, normaidők, szorzók, ügyfelek és ajánlatok relációs szerkezetben kezelhetők.

5. **UI:** Tailwind CSS vagy shadcn/ui
   Gyors, tiszta, modern adminfelületet lehet vele építeni.

6. **PDF ajánlat és munkalap:** React PDF vagy szerveroldali PDF generálás
   A végén az app tudjon ajánlatot, munkalapot vagy elszámolást exportálni.

## 2. Miért ezt választanám?

Az Ön céljára a **Next.js + Capacitor + Supabase** a legjobb kezdő irány.

Ennek oka egyszerű. A rendszer nem játékapp vagy nagy grafikai alkalmazás. Ez egy **adatvezérelt kalkulátor és adminrendszer**. Ehhez nem kell külön natív Android fejlesztés. Fontosabb a gyors fejlesztés, az adatbázis, az árlogika, az ajánlatkészítés és a későbbi bővíthetőség.

A Flutter is jó alternatíva, mert egy kódbázisból tud mobilra, webre és más platformokra is építeni. ([Flutter dokumentáció][4]) Viszont ebben az esetben a weboldali integráció és az admin jellegű felület általában egyszerűbben kezelhető Next.js irányból.

## 3. Az app fő logikája

Az alkalmazás lényege nem csak egy sima kalkulátor. Inkább egy **villanyszerelési normaidő- és díjkalkulátor**.

A számítás alapja:

**munkafolyamat mennyisége × normaidő × szorzók × rezsióradíj = munkadíj**

Példa:

1. Konnektor szerelése: 0,45 óra/db
2. Mennyiség: 12 db
3. Nehézségi szorzó: 1,2
4. Rezsióradíj: 14 000 Ft/óra

Számítás:

12 × 0,45 × 1,2 × 14 000 = 90 720 Ft

Ehhez még hozzáadható:

1. kiszállási díj
2. sürgősségi felár
3. bontási szorzó
4. régi épület szorzó
5. anyagköltség
6. kedvezmény
7. ÁFA
8. minimum díj

## 6. Számítási logika

A számításokat nem érdemes csak a frontendben tartani.

Jobb megoldás:

1. a frontend csak bekéri az adatokat
2. a backend számol
3. az adatbázis eltárolja az eredményt
4. a PDF ugyanabból az eltárolt adatból készül

Ez azért fontos, mert így nem változik meg utólag egy régi ajánlat, ha később módosítja a rezsióradíjat vagy a normaidőt.

A számítási képlet:

```text
alap_idő = mennyiség × normaidő

szorzott_idő = alap_idő × összes_szorzó

munkadíj = szorzott_idő × rezsióradíj

végösszeg = munkadíj + anyagköltség + kiszállás + ÁFA
```

## 11. Konkrét ajánlás

Én ezt választanám:

```text
Frontend: Next.js + React + TypeScript
UI: Tailwind CSS + shadcn/ui
Mobil: Capacitor Android
Backend: Supabase
Database: PostgreSQL
Auth: Supabase Auth
PDF: szerveroldali PDF generálás
Hosting: Vercel vagy Supabase
```

Ez a stack gyors, modern, költséghatékony, és jól illik egy villanyszerelési munkadíj-kalkulátorhoz. A legfontosabb döntés az, hogy a normaidők, rezsióradíjak és szorzók **adatbázisból vezéreltek** legyenek. Így később nem kell az appot újraprogramozni minden árváltozásnál.
